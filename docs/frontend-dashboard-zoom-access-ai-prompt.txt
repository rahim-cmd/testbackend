START_PROMPT

You are updating the frontend of the Circlia booking dashboard in a separate frontend repository.

Goal:
Update the user dashboard and admin screens to support dashboard-only Zoom access, password display, join window control, and admin enable/disable actions.

Important project context:
1) Backend and frontend are in separate repositories.
2) Backend is deployed separately from frontend.
3) Frontend is currently using a remote backend base URL.
4) Before testing locally, make sure the frontend API base URL points to the intended backend environment.
5) Do not hardcode production URLs.

Environment setup requirements:
1) Detect the existing frontend env variable used for API base URL.
2) Support local testing with: http://localhost:5000/api/v1
3) Preserve production configuration for deployed builds.
4) If the app uses Vite, likely env name is `VITE_API_URL` or similar.
5) If the app uses Next.js, likely env name is `NEXT_PUBLIC_API_URL` or similar.
6) Do not break existing auth token handling.

Backend rules:
1) Do not show a direct Zoom link in email.
2) Approved users must join from their Circlia dashboard only.
3) Join is enabled only within the meeting window.
4) Join should become available 2 minutes before the meeting start time.
5) The frontend must display the Zoom password to the user when the booking is approved.
6) If admin disables join access, the Join button must be disabled.
7) When user clicks Join, the backend logs the access and locks the booking join access until admin re-enables it.
8) User can end the session from the dashboard, but that does not re-enable join access.

API expectations:
1) GET /auth/profile
2) GET /bookings/my
3) POST /bookings/:id/join/start
4) POST /bookings/:id/join/end
5) GET /bookings/:id/join/logs
6) PUT /bookings/:id/join/control

Important backend response notes:
1) `meeting_date` now comes as a plain date string in `YYYY-MM-DD` format.
2) `approved_at`, `created_at`, `zoom_start_time`, and `zoom_updated_at` come as datetime strings.
3) `current_booking` is available on GET /auth/profile.
4) Dashboard may use either `/auth/profile` or `/bookings/my`, but it must correctly support the new fields from both.
